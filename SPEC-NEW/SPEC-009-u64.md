# SPEC-009: Camada de Chaves Canônicas, Mapeamento Denso e Representação Física para Índices

**Status:** CONGELADA / ALINHADA COM O CORE

**Fase do Workspace:** Milestones M1, M2 e M3

**Fórmula de Impacto:** Otimização de Layout em Memória e Eficiência Analítica de Cache

---

## 1. Desacoplamento Conceitual de Camadas

Para garantir que otimizações físicas de baixo nível não contaminem a semântica dos dados ou as garantias de auditoria e imutabilidade do log principal, o HeraclitusDB estabelece uma separação rigorosa em três níveis de abstração:

| Camada | Escopo | Responsabilidade | Estabilidade Semântica |
| --- | --- | --- | --- |
| **1. Logical Value** | Payloads do Log | Representa o dado bruto inserido pelo usuário (Strings, Atributos JSON, Assinaturas). | **Eterna.** Fonte da verdade absoluta do banco. |
| **2. Canonical Key** | Índices e Chaves | Representações `u64` canônicas temporárias geradas exclusivamente para fins de ordenação rápida. | **Volátil.** Existe apenas na fronteira dos índices. |
| **3. Physical Layout** | Estruturas Internas | A disposição mecânica dos bits na memória ou disco (Alinhamento em nós da Bᵉ-tree, arrays CSR de grafos). | **Mutável.** Altera de acordo com as políticas de compactação. |

Este desacoplamento impede vazamentos de abstração (*abstraction leaks*), isolando as mutações físicas decorrentes de ordenações de hardware dos identificadores globais do banco.

## 2. Alinhamento Crítico de Identidade (`EventId` e `DenseId`)

A auditoria do código-fonte e das especificações globais do workspace revelou uma inconsistência fundamental nas premissas de design iniciais que esta versão revisada corrige em caráter definitivo:

1. **Correção do Tamanho de `EventId`:** Rascunhos anteriores assumiam o `EventId` como um array binário de 32 bytes (Hash Blake3). No entanto, a especificação mestre (`SPEC.md`) e a implementação real do codec no H-VM (`heraclitus-core/src/vm/codec.rs`) estabelecem que o **`EventId` é um ULID estruturado de 128 bits (16 bytes)**, ordenado nativamente no tempo.
2. **Compactação de Identidade para `u32`:** Para otimizar ao máximo o empacotamento em linhas de cache, as visões materializadas locais (como o `GraphIndex` em `heraclitus-index-graph/src/lib.rs`) projetam esses identificadores de 16 bytes em sequências contíguas de inteiros de **32 bits (`u32`)** atribuidos de forma determinística na ordem do LSN.

### Impacto Mecânico na Cache Line da CPU

Em uma arquitetura moderna padrão com linhas de cache de 64 bytes:

* Armazenar identificadores globais `EventId` (ULID, 16 bytes) permite acomodar apenas **4 IDs** por linha de cache.
* Armazenar identificadores densos projetados localmente como `u32` (4 bytes) permite acomodar exatamente **16 IDs** por linha de cache.

Essa compressão mecânica de localidade representa um ganho de **400% de densidade volumétrica de cache**, reduzindo drasticamente os *cache misses* durante travessias longas efetuadas por operadores analíticos como o comando causal `WHY`.

## 3. Especificação do Codec de Chaves Canônicas (`CanonicalKeyCodec`)

O `CanonicalKeyCodec` opera na transição exata entre o valor lógico de payloads e chaves de índice estruturais. Ele assegura que a ordenação binária de inteiros sem sinal (`u64`) espelhe com perfeição a ordenação lexicográfica e numérica dos tipos nativos, incluindo números negativos e ponto flutuante IEEE-754.

> ⚠️ **Invariante de Normalização de NaNs e Zeros:** O método `encode_f64` realiza o colapso canônico de todas as variantes de NaN (*quiet NaNs*, *signaling NaNs* e payloads flutuantes distintos) na representação binária constante de teto `u64::MAX`. Adicionalmente, valores `-0.0` são colapsados em `+0.0` para blindar o índice contra duplicidades de chaves binárias para equivalências matemáticas neutras.

```rust
// heraclitus-core/src/vm/codec.rs

pub struct CanonicalKeyCodec;

impl CanonicalKeyCodec {
    pub const SIGN_BIT_MASK: u64 = 0x8000_0000_0000_0000;

    /// Transforma um i64 em u64 invertendo o bit de sinal (Flip Sign Bit)
    /// Garante que números negativos fiquem posicionados abaixo de números positivos na ordenação u64.
    #[inline]
    pub fn encode_i64(v: i64) -> u64 {
        (v as u64) ^ Self::SIGN_BIT_MASK
    }

    /// Reverte o u64 canônico para o i64 original
    #[inline]
    pub fn decode_i64(v: u64) -> i64 {
        v ^ Self::SIGN_BIT_MASK
    }

    /// Transforma um f64 em u64 canônico com colapso de NaN e normalização de zeros
    #[inline]
    pub fn encode_f64(v: f64) -> u64 {
        if v.is_nan() {
            return u64::MAX; // Todos os NaNs colapsam canonicamente no teto absoluto do índice
        }
        // Normaliza -0.0 para +0.0 evitando chaves binárias distintas
        let normalized_v = if v == 0.0 { 0.0 } else { v };
        let bits = normalized_v.to_bits();
        
        if (bits >> 63) == 0 {
            bits ^ Self::SIGN_BIT_MASK
        } else {
            !bits
        }
    }

    /// Reverte o u64 canônico para o f64 original
    #[inline]
    pub fn decode_f64(v: u64) -> f64 {
        if v == u64::MAX {
            return f64::NAN;
        }
        if (v & Self::SIGN_BIT_MASK) != 0 {
            f64::from_bits(v ^ Self::SIGN_BIT_MASK)
        } else {
            f64::from_bits(!v)
        }
    }
}

```

## 4. Ciclo de Vida Estrutural e Mapeamento Denso (`DenseEntityMap`)

A conversão de `EventId` lógicos para sequências numéricas compactas e locais segue um pipeline linear rigoroso de isolamento, dividido em três fases distintas para eliminar a necessidade de locks na engine analítica:

```
[ Fase 1: Replay ] ──> Ingestão linear e sequencial síncrona/concorrente em buffers mutáveis.
         │
         ▼
[ Fase 2: Optimize ] ──> Purga de tombstones, reordenação por afinidade física de cache.
         │
         ▼
[ Fase 3: Freeze ] ──> Conversão para arrays e visualizações congeladas imutáveis em Arc.

```

> 📜 **Contrato Arquitetural de Otimização:** O `DenseEntityMap` oculta propositalmente o algoritmo de hash utilizado internamente em sua fase mutável (ex: SwissTable/hashbrown) para evitar acoplamentos à especificação. Qualquer transformação ou renumeração física executada na fase *Optimize* deve sincronizar de forma coordenada as tabelas bidirecionais (`forward` e `backward`) antes de publicar a estrutura na Fase 3 (`Freeze`).

```rust
// heraclitus-index-graph/src/dense_map.rs

use std::sync::Arc;
use std::collections::hash_map::Entry;

pub type EventId = [u8; 16]; // Alinhado com o padrão ULID de 128 bits do Core

/// Fase 1: Estrutura mutável isolada otimizada para ingestão e Replay sequencial.
pub struct DenseEntityMap {
    // Utiliza uma abstração genérica de mapa hash otimizada para throughput durante o replay
    forward: std::collections::HashMap<EventId, u32>,
    backward: Vec<EventId>,
}

impl DenseEntityMap {
    pub fn with_capacity(capacity: usize) -> Self {
        Self {
            forward: std::collections::HashMap::with_capacity(capacity),
            backward: Vec::with_capacity(capacity),
        }
    }

    /// Aloca ou recupera um ID contíguo local mitigando double-hashing via API de Entry.
    pub fn get_or_alloc(&mut self, event_id: EventId) -> u32 {
        match self.forward.entry(event_id) {
            Entry::Occupied(e) => *e.get(),
            Entry::Vacant(e) => {
                let id = self.backward.len() as u32;
                self.backward.push(event_id);
                e.insert(id);
                id
            }
        }
    }

    /// Transiciona a estrutura aplicando as transformações sob contrato da Fase 2 e congelando na Fase 3.
    pub fn optimize_and_freeze(self) -> FrozenDenseEntityMap {
        // [Fase 2: Optimize] Espaço reservado para permutações de hardware e localidade física
        let final_forward = self.forward;
        let final_backward = self.backward;

        // [Fase 3: Freeze] Conversão total e imutável compartilhável sem locks entre threads analíticas
        FrozenDenseEntityMap {
            forward: Arc::new(final_forward),
            backward: Arc::from(final_backward),
        }
    }
}

/// Estado imutável estável: Zero locks, zero atomics, cache-line friendly para execução analítica.
pub struct FrozenDenseEntityMap {
    forward: Arc<std::collections::HashMap<EventId, u32>>,
    backward: Arc<[EventId]>,
}

impl FrozenDenseEntityMap {
    #[inline]
    pub fn lookup_id(&self, event_id: &EventId) -> Option<u32> {
        self.forward.get(event_id).copied()
    }

    #[inline]
    pub fn lookup_event(&self, id: u32) -> Option<EventId> {
        self.backward.get(id as usize).copied()
    }
}

```

## 5. Integração Física com o Modelo Matriz (Segmented CSR)

Os identificadores compactados em inteiros densos locais estabelecem as bases mecânicas essenciais para as estruturas descritas na `SPEC-010` (Compressed Sparse Row - CSR).

A conversão de referências globais esparsas (`EventId`) para coordenadas contíguas inteiras locais permite que as arestas causais de relacionamento sejam dispostas de forma vetorial homogênea na memória:

```
Vértice Denso (u32) ──> Projeção por Offset em Row Pointer (row_ptr) ──> Column Indices (col_idx)

```

Sem o mapeamento denso preliminar especificado nesta RFC, a criação de arranjos CSR compactos seria inviável, forçando o motor de grafos a retroceder para topologias baseadas em listas de ponteiros dispersos na Heap, destruindo a localidade espacial dos núcleos de execução analíticos.

## 6. Evolução de Hardware: Escaneamento Vetorizado Especializado (SIMD)

A padronização das chaves e delimitadores de intervalo em blocos homogêneos alinhados de chaves canônicas abre caminho para a injeção futura de kernels matemáticos paralelos em nível de silício.

A mera padronização do tipo não induz a aceleração imediata por hardware. Portanto, a arquitetura firma as seguintes obrigações técnicas de projeto para viabilizar vetorização real:

* **Alinhamento Estrito:** Estruturas de vetores e buffers de chaves ordinais congeladas na Fase 3 devem ser explicitamente alinhadas em memória seguindo fronteiras compatíveis com registradores vetoriais da arquitetura alvo (ex: alinhamento em 64 bytes para **AVX-512** / **AVX10** ou 16/32 bytes para **ARM NEON / SVE**).
* **Processamento por Kernels Especializados:** Operações de busca por faixas numéricas (`WHERE n.valor BETWEEN 1000 AND 5000`) serão processadas por loops abertos (*unrolled loops*) livres de desvios condicionais internos (*branchless code*). Isso permite que compiladores modernos ou assemblies especializados emitam instruções de máscara e varreduras paralelas brutas diretamente nos registradores vetoriais da CPU.

## 7. Critérios de Aceitação e Gates de Maturidade

* **Gate 1 (Invariância e Simetria):** Execução mandatória de 50.000 casos de teste via `proptests` validando a ordem lexicográfica estável de `CanonicalKeyCodec` para todos os números reais, bem como a perfeita reversibilidade pós-normalização de NaNs e sinalizadores de zero.
* **Gate 2 (Portabilidade Bit-a-Bit):** Garantia de reprodutibilidade cruzada idêntica de bits nas chaves canônicas geradas sob compilações nativas de arquiteturas distintas como `x86_64` e `AArch64`.
* **Gate 3 (Mitigação Analítica de Cache Misses):** O acoplamento de chaves canônicas com identificadores densos de 32 bits deve apresentar uma redução estatisticamente comprovada de falhas de cache L1/L2/L3 e *branch mispredictions* nos microbenchmarks automatizados do repositório em comparação com buscas efetuadas sobre os hashes esparsos nativos.
