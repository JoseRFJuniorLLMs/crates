# SPEC-009: Camada de Chaves Canônicas, Mapeamento Denso e Representação Física para Índices

## 1. Desacoplamento Conceitual de Camadas

Para garantir que otimizações físicas de baixo nível não contaminem a semântica dos dados ou as garantias de auditoria do log imutável, o HeraclitusDB estabelece uma separação rigorosa em três níveis de abstração:

| Camada | Escopo | Responsabilidade | Estabilidade Semântica |
| --- | --- | --- | --- |
| **1. Logical Value** | Payloads do Log | Representa o dado bruto inserido pelo usuário (Strings, Atributos JSON, Assinaturas).

 | **Eterna.** Fonte da verdade absoluta do banco.

 |
| **2. Canonical Key** | Índices e Chaves | Representações `u64` canônicas temporárias geradas exclusivamente para fins de ordenação rápida. | **Volátil.** Existe apenas na fronteira dos índices. |
| **3. Physical Layout** | Estruturas Internas | A disposição mecânica dos bits na memória ou disco (Alinhamento em nós da Bᵉ-tree, arrays CSR).

 | **Mutável.** Altera de acordo com as políticas de compactação. |

---

## 2. Especificação do Codec de Chaves Canônicas (`CanonicalKeyCodec`)

O `CanonicalKeyCodec` opera estritamente na transição entre o valor lógico e a chave de índice.

> ⚠️ **Nota de Engenharia (Normalização de NaNs):** O método `encode_f64` realiza o colapso canônico de todas as variantes de NaN (*quiet NaNs*, *signaling NaNs* e payloads flutuantes distintos) em uma única representação binária constante (`u64::MAX`). Portanto, a especificação garante a reversibilidade para todos os valores numéricos definidos, enquanto todas as representações de NaN são deliberadamente normalizadas para um único NaN canônico.

```rust
// heraclitus-core/src/vm/codec.rs

pub struct CanonicalKeyCodec;

impl CanonicalKeyCodec {
    pub const SIGN_BIT_MASK: u64 = 0x8000_0000_0000_0000;

    /// Transforma um i64 em u64 invertendo o bit de sinal (Flip Sign Bit)
    #[inline]
    pub fn encode_i64(v: i64) -> u64 {
        (v as u64) ^ Self::SIGN_BIT_MASK
    }

    /// Reverte o u64 canônico para o i64 original
    #[inline]
    pub fn decode_i64(v: u64) -> i64 {
        (v ^ Self::SIGN_BIT_MASK) as i64
    }

    /// Transforma um f64 em u64 canônico com colapso de NaN e normalização de zeros
    #[inline]
    pub fn encode_f64(v: f64) -> u64 {
        if v.is_nan() {
            return u64::MAX; // Todos os NaNs colapsam canonicamente no teto do índice
        }
        // Normaliza -0.0 para +0.0 evitando chaves binárias distintas
        let normalized_v = if v == 0.0 { 0.0 } else { v };
        let bits = normalized_v.to_bits();
        
        if (bits >> 63) == 0 {
            bits ^ Self::SIGN_BIT_MASK
        } else {
            !bits
        }
    }

    /// Reverte o u64 canônico para o f64 original
    #[inline]
    pub fn decode_f64(v: u64) -> f64 {
        if v == u64::MAX {
            return f64::NAN;
        }
        if (v & Self::SIGN_BIT_MASK) != 0 {
            f64::from_bits(v ^ Self::SIGN_BIT_MASK)
        } else {
            f64::from_bits(!v)
        }
    }
}

## 3. Ciclo de Vida Estrutural e Mapeamento Denso (`DenseEntityMap`)

A conversão de identificadores de 32 bytes (`EventId` do Blake3) para sequências numéricas compactas de 0 a $N$ segue um ciclo de vida linear dividido em três fases distintas:

```
[ Fase 1: Replay ] ──> Ingestão linear e sequencial sem travas.
         │
         ▼
[ Fase 2: Optimize ] ──> Remoção de tombstones, ordenação por localidade e renumeração.
         │
         ▼
[ Fase 3: Freeze ] ──> Conversão para tabelas estáticas imutáveis em Arc sem locks.

```

> 📜 **Contrato da Fase Optimize:** Toda e qualquer transformação aplicada aos identificadores durante a fase *Optimize* (como reordenação por afinidade de cache ou partições de comunidades) deve atualizar os índices de forma atômica e coordenada, preservando obrigatoriamente a estrita correspondência bidirecional entre os mapas `forward` e `backward` antes da geração final das estruturas congeladas.

```rust
// heraclitus-index-graph/src/dense_map.rs

use fxhash::{FxHashMap, FxBuildHasher};
use std::collections::hash_map::Entry;
use std::sync::Arc;

pub type EventId = [u8; 32]; // Hash Blake3 do HeraclitusDB[cite: 2]

/// Fase 1: Estrutura puramente mutável otimizada para o Replay linear e sequencial
pub struct DenseEntityMap {
    forward: FxHashMap<EventId, u64>,
    backward: Vec<EventId>,
}

impl DenseEntityMap {
    pub fn with_capacity(capacity: usize) -> Self {
        Self {
            forward: FxHashMap::with_capacity_and_hasher(capacity, FxBuildHasher::default()),
            backward: Vec::with_capacity(capacity),
        }
    }

    /// Insere ou recupera um ID contíguo usando a API de Entry para mitigar double-hashing
    pub fn get_or_alloc(&mut self, event_id: EventId) -> u64 {
        match self.forward.entry(event_id) {
            Entry::Occupied(e) => *e.get(),
            Entry::Vacant(e) => {
                let id = self.backward.len() as u64;
                self.backward.push(event_id);
                e.insert(id);
                id
            }
        }
    }

    /// Fase 2 & 3: Ponto de otimização mecânica e congelamento absoluto livre de atomics/locks
    pub fn optimize_and_freeze(self) -> FrozenDenseEntityMap {
        // [Fase 2: Optimize] Espaço reservado para transformações sob contrato de invariância
        let final_forward = self.forward;
        let final_backward = self.backward;

        // [Fase 3: Freeze] Congelamento total de ambas as coleções em estruturas imutáveis em Arc
        FrozenDenseEntityMap {
            forward: Arc::new(final_forward),
            backward: Arc::from(final_backward),
        }
    }
}

/// Estado imutável final: Zero locks, zero atomics, cache-friendly absoluto para a Engine analítica
pub struct FrozenDenseEntityMap {
    forward: Arc<FxHashMap<EventId, u64>>,
    backward: Arc<[EventId]>,
}

impl FrozenDenseEntityMap {
    #[inline]
    pub fn lookup_id(&self, event_id: &EventId) -> Option<u64> {
        self.forward.get(event_id).copied()
    }

    #[inline]
    pub fn lookup_event(&self, id: u64) -> Option<EventId> {
        self.backward.get(id as usize).copied()
    }
}

## 4. Evolução Futura: Escaneamento Vetorizado via Hardware (SIMD)

A padronização das chaves de indexação e chaves de intervalo da Bᵉ-tree em arrays homogêneos de `u64` abre o caminho para o uso futuro de instruções vetoriais paralelas (`SIMD Range Scanning`).

Quando implementados kernels especializados alinhados em memória contígua e em blocos de tamanho fixo, filtros com restrições numéricas (`WHERE n.valor BETWEEN 1000 AND 5000`) poderão carregar múltiplos blocos de chaves `u64` diretamente em registradores vetoriais (**AVX-512** ou **NEON**), executando varreduras brutas diretamente no silício da CPU sem desvios condicionais custosos.

## 5. Critérios de Aceitação (Gates de Maturidade)

* **Gate 1 (Invariância e Simetria):** Execução de 50.000 casos via `proptests` validando a integridade lexicográfica para todos os números reais e garantindo a reversibilidade estável para valores definidos após a normalização de NaNs e sinalizadores de zero.
* **Gate 2 (Portabilidade de Bits):** Validação cruzada idêntica bit a bit das chaves canônicas produzidas em compilações nativas para arquiteturas `x86_64` e `AArch64`.
* **Gate 3 (Métricas de Eficiência de Cache):** A camada de chaves canônicas e identificadores densos deve apresentar redução estatisticamente significativa nas métricas de falhas de cache L1/L2 e *branch mispredictions* tomando como referência as cargas de trabalho estáveis nos benchmarks oficiais do repositório do HeraclitusDB.

# 📑 SPEC-009: Camada de Chaves Canônicas, Mapeamento Denso e Engenharia de Execução Analítica

## 1. Desacoplamento Conceitual de Camadas

Para garantir que otimizações físicas de baixo nível não contaminem a semântica dos dados ou as garantias de imutabilidade do log, o HeraclitusDB estabelece uma separação rigorosa em três níveis de abstração:

| Camada | Escopo | Responsabilidade | Estabilidade Semântica |
| --- | --- | --- | --- |
| **1. Logical Value** | Payloads do Log | Representa o dado bruto inserido pelo usuário (Strings, Atributos JSON, Assinaturas). | **Eterna.** Fonte da verdade absoluta do banco. |
| **2. Canonical Key** | Índices e Chaves | Representações `u64` canônicas temporárias geradas exclusivamente para fins de ordenação rápida. | **Volátil.** Existe apenas na fronteira dos índices. |
| **3. Physical Layout** | Estruturas Internas | A disposição mecânica dos bits na memória ou disco (Layouts de matrizes esparsas, alinhamento de nós da árvore). | **Mutável.** Altera de acordo com as políticas de compactação. |

---

## 2. Ciclo de Vida Estrutural: O Pipeline Core

Toda e qualquer otimização analítica ou física aplicada às visões derivadas do HeraclitusDB ocorre de forma isolada, respeitando o pipeline linear e unidirecional de três fases:

```
[ Fase 1: Replay ] ──> Ingestão linear e sequencial (Estruturas mutáveis otimizadas para escrita).
         │
         ▼
[ Fase 2: Optimize ] ──> Remoção de tombstones, reordenação por localidade e renumeração física.
         │
         ▼
[ Fase 3: Freeze ] ──> Conversão para layouts estáticos, imutáveis e thread-safe para leitura.

```

## 3. Desacoplamento de Identificadores (Mapeamento de Dupla Indireção)

Para que a fase `Optimize` possa reorganizar dados fisicamente na memória sem quebrar referências externas ou caches de alto nível, o sistema de mapeamento denso rejeita atribuições físicas diretas e estabelece uma camada de **dupla indireção**:

```
[EventId (32B)] ──> [Stable ID (u64)] ── (Fase Optimize) ──> [Physical Dense ID (u64)]

```

* **`Stable ID`**: Sequência numérica alocada de forma estritamente determinística durante a fase de `Replay`. Serve como âncora para filtros relacionais externos e planos lógicos.
* **`Physical Dense ID`**: Identificador volátil que dita a coordenada exata (linha/coluna) das matrizes de adjacência. Pode ser livremente renumerado para otimizar afinidade de cache e agrupamentos de blocos.

> 🛡️ **Invariante Crítica de Identidade:** Um `StableId` nunca pode ser reutilizado para representar outro `EventId`, mesmo após compactação de dados, remoção de tombstones ou reconstrução total de índices físicos.

---

## 4. Representação Física Adaptativa: Segmented & Hyper-Sparse CSR

O formato físico de indexação relacional descarta listas de adjacência baseadas em mapas mutáveis concorrentes para leitura e assume o modelo de **Segmented CSR (Compressed Sparse Row)**.

### A. Segmentação Linear por Log Intervals

Como consequência direta da filosofia *append-only* do banco, o espaço global de relacionamentos é fatiado em **Segmentos CSR** imutáveis independentes. Cada segmento representa uma janela fechada do log. Durante o `Freeze`, o segmento ativo é consolidado em uma estrutura física estática, eliminando a necessidade de reconstruir globalmente as matrizes de adjacência a cada escrita.

### B. Polimorfismo Físico por Densidade

A estrutura de cada segmento é dinâmica. Durante a transição na fase `Optimize`, o mecanismo de armazenamento avalia a densidade de arestas e seleciona automaticamente o layout físico mais eficiente:

* **CSR Tradicional**: Escolhido para subgrafos densos, minimizando indireções.
* **Hyper-Sparse CSR**: Escolhido quando a quantidade de linhas vazias excede o limite aceitável (comum em cadeias causais estritas), introduzindo um array intermediário `non_empty_rows` para impedir o desperdício de memória com ponteiros nulos.

---

## 5. Interface de Operadores Físicos (`Physical Operator Interface`)

Para que o motor analítico (`heraclitus-analytics`) possa executar consultas complexas sem se acoplar a uma estratégia física fixa, introduzimos a abstração de operadores físicos via *Traits*. O planejador manipula apenas essa interface abstrata.

```rust
pub struct ExecutionContext {
    pub target_lsn: u64,
    pub session_flags: u32,
}

pub enum OperatorResult {
    IntermediaryGraph(Vec<u64>), // Vetor de Stable IDs
    ColumnarBatch,                // Interoperabilidade com o formato do log analítico
}

pub trait GraphOperator {
    /// Executa a operação analítica sobre os dados físicos isolados
    fn execute(&self, ctx: &ExecutionContext) -> Result<OperatorResult, String>;
}

```

## 6. Mecanismo de Planejamento e Catálogo de Estatísticas

A escolha de qual estratégia física adotar para responder a uma consulta (ex: travessia imperativa ou cálculo matricial) é uma responsabilidade em camadas independentes:

### A. Catálogo de Estatísticas Persistentes (`Statistics Catalog`)

Componente conceitual que armazena metadados estatísticos computados e atualizados ao final de cada fase `Freeze`. O catálogo expõe métricas consolidadas, eliminando a necessidade de varreduras caras em tempo de execução:

* Grau médio e distribuição de graus de conectividade.
* Métricas de densidade global e por segmento.
* Histogramas de seletividade e estimativas de cardinalidade.

### B. O Fluxo do Planejador (Query Execution Pipeline)

O fluxo de resolução de uma consulta segue a separação estrita de responsabilidades:

1. **Statistics**: Coleta métricas físicas reais estabilizadas no catálogo.
2. **Cardinality Estimation**: Estima o volume de dados que passará por cada junção ou salto causal.
3. **Cost Model**: Avalia o custo de computação (CPU vs I/O vs Cache Locality) para as diferentes abordagens disponíveis.
4. **Physical Planner**: Instancia a cadeia de objetos que implementam o `GraphOperator` adequado.
5. **Execution**: Dispara o processamento físico no hardware.

* *Cenário de Baixa Cardinalidade (Buscas Locais)*: O modelo de custo detecta que o operador afetará poucos vértices e despacha um operador de **Busca Imperativa Clássica**.
* *Cenário de Alta Cardinalidade (Varreduras de Multi-Saltos Globais)*: O modelo de custo detecta grandes volumes de dados e despacha um operador baseado no **Sparse Linear Algebra Engine** (usando a implementação de referência do *GraphBLAS* ou aceleradores equivalentes).

## 7. Acoplamento de Relacionamentos: Multi-Relation Matrices

O layout físico isola os diferentes tipos de conexões do log em estruturas de matrizes independentes e ortogonais. Relacionamentos biográficos (`parents`), temporais, dependências lógicas e correlações estatísticas geram segmentos esparsos separados (ex: $A_{parent}$, $A_{temporal}$).

Isso permite que buscas complexas sejam resolvidas através da composição de operações de álgebra linear esparsa encadeadas no silício:


$$\vec{v}_{next} = \vec{v}_{start} \times A_{parent} \times A_{temporal}$$

## 8. Invariantes Mecânicas de Hardware (Agnosticismo Tecnológico)

Para garantir que esta RFC envelheça com resiliência diante da evolução dos processadores, as otimizações de hardware são descritas por contratos de comportamento, tratando tecnologias atuais como implementações de referência:

* **NUMA Affinity**: O layout físico e o alocador de memória associado aos segmentos congelados devem, obrigatoriamente, preservar e honrar a afinidade de nó de memória (*memory affinity policies*) onde as threads de execução estão ativas. Isso impede a degradação de throughput por cruzamento de barramentos físicos entre sockets de CPU.
* **Vectorial Acceleration Engine**: Os kernels matemáticos de varredura devem estruturar os arrays contíguos de chaves canônicas e coordenadas de matrizes de forma alinhada à arquitetura de cache (*cache-line aware*). O código deve ser estruturado de forma a permitir que o compilador ou os kernels especializados invoquem **instruções vetoriais paralelas suportadas pela arquitetura alvo** (sejam elas extensões do tipo AVX-512, AVX10, ARM SVE, RISC-V Vector Extensions ou aceleradores dedicados de GPU).
