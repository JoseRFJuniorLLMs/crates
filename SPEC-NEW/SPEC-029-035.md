# 📑 SPEC-029: Manifesto de Compatibilidade Binária e Versionamento de Formato (`Storage Format Versioning`)

## 1. Escopo Conceitual

Esta especificação impede que atualizações nas estratégias de compressão adaptativa ou layouts colunares corrompam dados históricos ou causem quebras silenciosas em clusters distribuídos. Estabelece-se um protocolo estrito de negociação de capacidades binárias na fronteira de inicialização do disco.

## 2. Estrutura do Cabeçalho de Formato

Todo arquivo de segmento autocontido e o próprio `DatabaseManifest` devem iniciar com a assinatura rígida de layout:

```rust
pub struct StorageFormatVersion {
    pub major: u16,          // Alterações que quebram retrocompatibilidade (ex: mudança estrutural do log)
    pub minor: u16,          // Adições retrocompatíveis (ex: novo codec de compressão adaptativa)
    pub feature_flags: u64,  // Bitmask indicando recursos ativados (ex: bit 0 = HyperSparse, bit 1 = RibbonFilter)
}

```

## 3. Protocolo de Negociação e Rejeição

1. **Fronteira de Leitura**: Ao carregar um segmento, o `StorageEngine` valida a tripla de versão contra as capacidades compiladas no binário atual.
2. **Incompatibilidade Avançada (*Forward Incompatibility*)**: Se o campo `major` for superior ao suportado pelo nó, ou se a `feature_flags` contiver um bit ativo desconhecido pelo executável, a inicialização é abortada imediatamente com erro fatal de pânico de formato, impedindo a degradação ou interpretação errada de bytes.
3. **Evolução Segura**: Se `minor` for superior, mas `major` coincidir, o nó ignora os metadados desconhecidos secundários e prossegue com o replay bruto do log, que permanece idêntico.

# 📑 SPEC-030: Regras Formais de Execução Determinística para Replay Multi-Thread (`Deterministic Replay Execution Rules`)

## 1. Escopo Conceitual

A premissa de que o replay distribuído e paralelo gera resultados logicamente idênticos quebra se o silício aplicar otimizações aritméticas arbitrárias. Esta especificação dita as restrições mecânicas que anulam qualquer divergência binária entre diferentes compiladores ou microarquiteturas (`x86_64` vs `AArch64`).

## 2. Invariantes Aritméticas e de Ordenação

### A. Ordem Canônica de Merge

As threads que processam segmentos históricos fora de ordem geram arrays de adjacência parciais. O processo de combinação final (*Merge Engine*) é obrigado a ordenar as chaves físicas de destino usando uma **função de ordenação lexicográfica estável bit a bit**, anulando qualquer variação decorrente do agendamento de threads do sistema operacional.

### B. Proibição de Redução Associativa em Ponto Flutuante

Operações de ponto flutuante (`f64`) não são puramente associativas devido a arredondamentos de bits de precisão no padrão IEEE 754.

* **A Regra**: É terminantemente proibido realizar somas acumuladas ou reduções paralelas de floats onde a ordem dos operandos mude com base na velocidade das threads.
* **A Solução**: O HeraclitusDB impõe que qualquer acumulação analítica colunar use uma **Árvore de Redução Fixa Determinística** ou execute o colapso sequencial estrito indexado pelo `Stable ID`. Alternativamente, o modelo de custo prioriza o escalonamento via inteiros escalados (*fixed-point arithmetic*).

# 📑 SPEC-031: Grafo de Dependências e Cascata de Invalidação de Artefatos (`Artifact Dependency DAG`)

## 1. Escopo Conceitual

No modelo JIT de estruturas do HeraclitusDB, os artefatos efêmeros nascem de forma encadeada. Se uma matriz esparsa CSR depende de um filtro de bits colunar, e esse filtro é limpo da memória pelo `MemoryManager`, a matriz CSR torna-se um objeto inválido. Esta especificação introduz um DAG (Directed Acyclic Graph) de gerenciamento interno de ciclo de vida de memória.

## 2. O Registro de Vínculos de Dependência

O `ArtifactRegistry` passa a rastrear a árvore de derivação física:

```rust
pub struct ArtifactDependencyNode {
    pub artifact_id: u64,
    pub depends_on: Vec<u64>,  // Lista de IDs de artefatos dos quais este objeto depende diretamente
    pub downstream_children: Vec<u64>, // Artefatos que dependem deste nó
}

```

## 3. Protocolo de Evicção em Cascata (`Eviction Cascade`)

Quando o `MemoryManager` decide expurgar um artefato da RAM devido à alta pressão de memória, o fluxo segue o protocolo de segurança do DAG:

```
[ Comando de Desalocação: ZoneMap A ]
                │
                ▼
      Varre downstream_children
                │
                ├────────────────────────┐
                ▼                        ▼
      [ Invalida CSR Matrix ]   [ Invalida HNSW Cache ]
                │                        │
                ▼                        ▼
        Destrói da RAM           Destrói da RAM

```

**Invariante de Segurança**: Nenhum artefato órfão pode permanecer na memória. A destruição de um nó raiz força a limpeza imediata e recursiva de toda a sua árvore descendente, prevenindo ponteiros corrompidos ou leituras de visões desatualizadas.

# 📑 SPEC-032: Malha de Retroalimentação Estatística do Otimizador (`Adaptive Execution Feedback`)

## 1. Escopo Conceitual

Esta especificação remove a natureza estática do modelo de custo de consultas analíticas. Introduz-se uma malha de controle de retroalimentação endógena que permite ao planejador aprender com seus próprios erros de estimativa, ajustando dinamicamente os limiares de decisão através de estatísticas puras, sem o uso de Inteligência Artificial generativa ou redes neurais complexas.

## 2. A Métrica de Desvio de Predição

A cada execução final de um `GraphOperator`, a engine captura a telemetria real de tempo de CPU e acessos a cache e injeta no sistema o registro de feedback:

```rust
pub struct ExecutionFeedback {
    pub query_fingerprint: [u8; 32],
    pub estimated_cost_score: u64,
    pub actual_execution_time_ns: u64,
    pub prediction_error_ratio: f32, // (Real - Estimado) / Estimado
}

```

## 3. Calibração via Média Móvel Exponencial (EMA)

O `StatisticsCatalog` consome esses feedbacks e atualiza os coeficientes de peso das equações de custo usando uma média móvel exponencial. Se o planejador estimou que uma multiplicação esparsa de matrizes demoraria 10ms, mas a concorrência de hardware causou uma latência real de 200ms, o erro inflaciona o peso desse operador no catálogo. Nas próximas consultas semelhantes, o planejador escolherá automaticamente o fallback para buscas imperativas locais.

# 📑 SPEC-033: Governança Física de Memória em Arquiteturas NUMA (`NUMA Memory Governance`)

## 1. Escopo Conceitual

Em servidores multiprocessados grandes, cruzar o barramento de interconexão física de CPU para ler dados localizados em um socket de RAM remoto destrói o throughput da álgebra linear esparsa. Esta especificação dita as regras de afinidade e movimentação de estruturas analíticas contíguas através de nós NUMA.

## 2. Invariantes de Alocação e Políticas de Localidade

### A. Afinidade de Thread-Nodo

O `ResourceScheduler` obriga que as threads de computação matemática do GraphBLAS sejam vinculadas (*core pinning*) aos núcleos que compartilham a cache L3 local com o nó de memória onde os metadados colunares e segmentos CSR correspondentes foram mapeados.

### B. Arbitragem de Acesso Transnodo

Se uma consulta disparada na CPU do `Nodo 0` necessitar processar um `DerivedExecutionArtifact` alocado na memória do `Nodo 1`, o modelo de custo aplica uma regra polimórfica baseada no tamanho da estrutura:

* **Estratégia de Replicação**: Se o artefato for pequeno (ex: um bitmap de filtro), o sistema duplica o objeto diretamente no `Nodo 0`.
* **Estratégia de Recompilação Local**: Se for uma matriz esparsa CSR gigantesca, transferir os gigabytes via barramento causaria alta latência. O sistema invoca o **JIT Compiler** local e dispara um replay parcial assíncrono paralelo nas threads do `Nodo 0`, reconstruindo a matriz localmente a partir da cache de leitura do log de episódios, preservando a velocidade mecânica pura do silício.

# 📑 SPEC-034: Modelo Concorrente de Ciclo de Vida de Memória (`Memory Lifetime Model`)

## 1. Escopo Conceitual

Esta especificação formaliza a estratégia de gerenciamento de memória em nível de código Rust para satisfazer o *Borrow Checker* sem introduzir contenção de travas globais de escrita em estruturas concorrentes complexas, garantindo vazamento zero de memória.

## 2. Primitivas de Gerenciamento Concorrente

O HeraclitusDB adota uma arquitetura híbrida de gerenciamento de referências baseada nas três fases do pipeline core:

| Componente / Estrutura | Tipo de Smart Pointer / Mecanismo | Justificativa de Engenharia |
| --- | --- | --- |
| **Fase 1: Buffers de Replay Ativos** | `DashMap` + `Arc<RwLock<T>>` | Permite múltiplas escritas e appends lineares concorrentes isolados por shards de baldes. |
| **Fase 3: Segmentos CSR Congelados** | `Arc<[u64]>` | Arrays estáticos puros contíguos na memória de leitura, compartilhados entre queries com overhead zero. |
| **Artefatos Efêmeros (`Transient`)** | `Weak<DerivedExecutionArtifact>` | Permite que o `ArtifactRegistry` mantenha o catálogo do objeto sem impedir sua destruição imediata pelo gerenciador. |
| **Varredura Analítica Sem Travas** | **Epoch-Based Reclamation (EBR)** | Garante que leitores leiam matrizes antigas com segurança enquanto a fase Optimize consolida a nova versão. |

## 3. O Mecanismo de Reclamação Baseado em Épocas (EBR)

Para evitar o uso de contadores de referência atômicos pesados (`Arc::clone`) dentro dos loops internos críticos de multiplicação de matrizes esparsas:

1. As threads de leitura analítica entram em uma "Época" global protegida ao iniciar a consulta.
2. Se o pipeline de manutenção gerar uma matriz CSR otimizada revisada e substituir o ponteiro global, a matriz CSR antiga não é desalocada da RAM imediatamente.
3. Ela é movida para uma fila de lixo temporária e destruída de forma segura somente quando todas as threads ativas na época de leitura antiga saírem do contexto de execução, mitigando completamente o risco de falhas de segmentação ou uso de memória após a liberação (*use-after-free*).

# 📑 SPEC-035: Isolamento de Componentes de Extensão via Sandboxing WebAssembly (`Plugin Sandboxing & Stable ABI`)

## 1. Escopo Conceitual

Mecanismos de extensibilidade tradicionais que carregam bibliotecas dinâmicas compartilhadas (`.so` / `.dll`) diretamente no espaço de endereçamento do processo principal são perigosos. Se um plugin de compressão de terceiros contiver um erro de ponteiro ou estourar a pilha, ele derrubará o banco de dados inteiro. Esta especificação blinda o HeraclitusDB isolando extensões dentro de uma sandbox rígida com interface de aplicação estável (ABI).

## 2. Arquitetura de Isolamento com Sandbox WASM

Todos os plugins cadastrados no `ExtensionCapabilities` são carregados e executados exclusivamente dentro de um runtime **WebAssembly (WASM)** embarcado (utilizando a implementação de referência do `Wasmtime` ou `Extism`).

```
[ HeraclitusDB Core Process ]
              │
              ▼ (Stable ABI Call)
┌────────────────────────────────────────┐
│ WASM Sandbox Memory Isolation Boundary │
│   └── Third-Party Extension Plugin     │ -> Executa compressão ou índice customizado.
└────────────────────────────────────────┘

```

## 3. Restrições e a ABI Estável

* **Isolamento de Heap**: O plugin opera em um espaço linear de memória totalmente isolado. Ele não possui ponteiros diretos para os arrays CSR do core ou para o sistema de arquivos do host.
* **Overhead Controlado**: A troca de dados entre o processo em Rust e a sandbox ocorre via passagem de ponteiros de memória linear serializados em vetores Arrow contíguos.
* **Blindagem de Crash**: Se o plugin sofrer um pânico interno, estourar a memória permitida na cota ou entrar em um loop infinito, o runtime do HeraclitusDB intercepta a falha como um erro clássico tratado de execução, revoga as credenciais da extensão e descarta a instância da sandbox sem afetar o log imutável ou interromper as consultas dos usuários ativos.

### 🏁 Matriz Final de Prontidão de Engenharia (Gates de Maturidade)

Com a incorporação das resoluções das SPECs 029 a 035, o plano conceitual atinge sua plenitude operacional absoluta:

1. **Gate 1 (Blindagem de Identidade)**: Dupla indireção de IDs estáveis contra renumeração física eliminada como fonte de bugs.
2. **Gate 2 (Governança Mecânica)**: Alocação NUMA-aware e gerenciamento concorrente EBR formalizados no nível do silício.
3. **Gate 3 (Estabilidade de Execução)**: Sandboxing WASM impede que o ecossistema de extensões de terceiros degrade ou cause pânicos no processo principal do core.
