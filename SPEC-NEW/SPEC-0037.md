# SPEC-0037: Infraestrutura de Execução Analítica Avançada e Federação (SOTA Blueprint)

A especificação `SPEC-2026-HERACLITUS` consolidou um motor estável, seguro e resiliente para o ambiente corporativo. No entanto, para posicionar o **Heraclitus** no topo absoluto do desempenho analítico global (**State-of-the-Art - SOTA**), a arquitetura precisa transcender a barreira dos modelos estáticos.

Este documento de engenharia estabelece as fundações para reestruturar o motor de execução, transformando-o em um ecossistema adaptativo, compilado por JIT, orientado a pipelines de dados e nativamente federado.

---

## 1. Adaptive Query Execution (AQE)

O Otimizador Baseado em Custos (CBO) estático é vulnerável a erros catastróficos de estimativa devido a predicates complexos, correlação entre colunas e dados não uniformes. O Heraclitus adotará a execução adaptativa dividindo o plano físico em estágios isolados por barreiras de materialização (*Query Stages*).

```
[ Query Stage 1 (Scan & Filter) ] ➔ Materializa Intermediário (Morsel)
                                             │
                                             ▼
                              [ Estatísticas em Tempo Real ]
                                             │
                                             ▼
[ Otimizador Dinâmico (AQE) ] ➔ Replaneja o restante da árvore de execução
                                             │
                                             ▼
[ Query Stage 2 (Dynamic Join Strategy: Hash ➔ Broadcast) ]

```

### Mecanismos de Replanejamento em Runtime

* **Coalescência Dinâmica de Partições**: Se após um operador de filtro o volume de dados por partição cair drasticamente, o AQE agrupa partições adjacentes em tempo de execução para mitigar o overhead de agendamento de tarefas pequenas.
* **Alternância Dinâmica de Joins**: Se a cardinalidade real coletada na barreira de materialização de um lado da junção for inferior ao limite de memória do operador `Exchange`, o plano físico altera dinamicamente um *Sort-Merge Join* ou *Shuffled Hash Join* para um *Broadcast Hash Join*, eliminando o tráfego de rede da maior tabela.
* **Otimização de Skew (Assimetria)**: Detecção de partições desproporcionalmente grandes em tempo de execução. O AQE dividirá automaticamente a partição assimétrica em subpartições, duplicando o lado oposto da junção para paralelizar o processamento da assimetria sem causar gargalos em uma única thread.

## 2. Cost Model Moderno e Multidimensional

O modelo de custo linear baseado puramente em número de tuplas é obsoleto para hardwares modernos com paralelismo massivo e hierarquias complexas de cache. O arquivo `cost.rs` será reescrito para computar um vetor de custo multidimensional e contínuo.

A equação do custo total ponderado do plano de execução é definida por:

$$\text{Custo}_{\text{total}} = \psi_{\text{cpu}} \cdot \mathcal{C}_{\text{cpu}} + \psi_{\text{mem}} \cdot \mathcal{C}_{\text{mem}} + \psi_{\text{io}} \cdot \mathcal{C}_{\text{io}} + \psi_{\text{net}} \cdot \mathcal{C}_{\text{net}}$$

Onde as componentes analíticas do vetor de custo são detalhadas a seguir:

### Custo de Computação ($\mathcal{C}_{\text{cpu}}$)

$$\mathcal{C}_{\text{cpu}} = \sum_{i \in \text{ops}} N_i \cdot \left( \frac{\text{Ciclos}_{\text{op}}}{\text{Fator}_{\text{SIMD}}} + \text{Overhead}_{\text{decomp}} \right)$$


Onde $N_i$ é a cardinalidade estimada, $\text{Ciclos}_{\text{op}}$ representa o custo intrínseco da instrução e $\text{Fator}_{\text{SIMD}}$ mensura a capacidade de vetorização do tipo de dado na CPU alvo.

### Custo de Memória e Cache ($\mathcal{C}_{\text{mem}}$)

$$\mathcal{C}_{\text{mem}} = \sum_{j \in \text{scans}} \left( \text{Misses}_{\text{L1/L2}} \cdot \text{Lat}_{\text{L3}} + \text{Misses}_{\text{L3}} \cdot \text{Lat}_{\text{RAM}} \right) + \mu_{\text{alloc}} \cdot \text{RAM}_{\text{footprint}}$$


Estima o impacto do layout de memória das tabelas de dispersão (*Hash Tables*) contra falhas de cache de CPU baseadas no tamanho das linhas de cache do sistema.

### Custo de Transferência e Latência de IO ($\mathcal{C}_{\text{io}}$ e $\mathcal{C}_{\text{net}}$)

$$\mathcal{C}_{\text{io}} = \frac{\text{Bytes}_{\text{disco}}}{\text{Bandwidth}_{\text{NVMe}}} + \text{Lat}_{\text{seek}}, \quad \mathcal{C}_{\text{net}} = \frac{\text{Bytes}_{\text{rede}}}{\text{Bandwidth}_{\text{network}}} + \text{Lat}_{\text{network}}$$


Distingue de forma agressiva acessos sequenciais a blocos colunares via pré-carregamento assíncrono de leituras aleatórias para busca de texto ou caminhamento de grafos.

## 3. Runtime Pipeline Execution Engine

O modelo clássico de iteradores (Volcano-style, `next()`) destrói a localidade de cache de instrução e introduz um overhead massivo de chamadas de função virtuais. O Heraclitus migrará para uma arquitetura orientada a pipelines baseada em processamento por blocos direcionados por vetores (*Morsel-Driven Parallelism*).

```
                       [ PIPELINE DE EXECUÇÃO (Push-Based) ]
                                         │
 ┌───────────────────────────────────────┴───────────────────────────────────────┐
 ▼                                       ▼                                       ▼
[ Morsel 1 (100k linhas) ]      [ Morsel 2 (100k linhas) ]      [ Morsel 3 (100k linhas) ]
 │                               │                               │
 ▼                               ▼                               ▼
[ Operador: Vector Scan ]       [ Operador: Vector Scan ]       [ Operador: Vector Scan ]
 │                               │                               │
 ▼                               ▼                               ▼
[ Operador: Vector Filter ]     [ Operador: Vector Filter ]     [ Operador: Vector Filter ]
 │                               │                               │
 └───────────────────────────────────────┬───────────────────────────────────────┘
                                         ▼
                        [ Pipeline Breaker: Hash Build ]

```

### Arquitetura de Execução Física

* **Push-Based Execution**: Em vez de operadores filhos puxarem dados dos pais, os operadores de origem empurram vetores de dados fixos (ex: 102.400 registros por bloco) através de cadeias lineares de operadores em linha reta até atingirem um ponto de sincronização ou materialização (*Pipeline Breaker*, como a construção de uma Hash Table).
* **Morsel-Driven Scheduling**: As threads de execução tornam-se agnósticas a partições rígidas. O escalonador distribui dinamicamente fragmentos de memória (*Morsels*) de forma coordenada por localidade NUMA, garantindo que os núcleos da CPU operem continuamente em loops analíticos apertados.

## 4. JIT Compilation Tier (Runtime Compilado)

Embora a máquina virtual contida em `heraclitus-core` seja excelente para isolamento de UDFs e extensibilidade de plugins, loops de avaliação de expressões complexas em consultas puramente analíticas introduzem overhead de decodificação de bytecode. Introduziremos uma camada de compilação Just-In-Time (JIT) opcional focada em operadores críticos utilizando o compilador **Cranelift** (por sua velocidade de compilação ultra-rápida em Rust).

### Estratégia Híbrida de Compilação

O planejador avalia a complexidade da consulta. Se o tempo estimado de execução superar o limiar crítico de compilação (ex: processamento de dezenas de milhões de linhas), o Heraclitus compila as subárvores de expressões matemáticas, projeções e filtros diretamente em código de máquina nativo.

```rust
// Exemplo conceitual da geração de código nativo para um operador de Filtro+Projeção
fn compile_morsel_pipeline(expression: &Expr) -> TargetCode {
    let mut ctx = CraneliftContext::new();
    // Transforma a AST da consulta em IR do Cranelift
    ctx.emit_vector_loop_instructions(expression);
    ctx.finalize_native_function()
}

```

O código compilado opera diretamente sobre ponteiros de memória de arrays Arrow ou blocos colunares, eliminando ramificações de decisão (*branches*) dentro de loops internos de processamento de dados.

## 5. Adaptive Compression Strategy

Em vez de forçar o uso de um método fixo por tipo de coluna ou baseado em amostragem estática na primeira ingestão, o componente `heraclitus-columnar` passará a avaliar de forma dinâmica e contínua a entropia e distribuição dos dados por página física de escrita.

### Algoritmos e Regras Dinâmicas por Página

O motor analisa o bloco de dados no Buffer antes da descarga física para o disco ou Object Storage:

* **Low Cardinality Detectada**: O motor converte automaticamente a página para **RLE (Run-Length Encoding)** ou **Dictionary Encoding**.
* **Monotonically Increasing/Sequential (Timestamps, IDs)**: Aplica-se **Delta Encoding** combinado com **Bit Packing**.
* **Bounded Range Integers**: Reduz a representação de bits usando **FOR (Frame of Reference)**.

A estratégia escolhida é registrada no cabeçalho da página individual. O decodificador vetorizado SIMD lê esse cabeçalho e chaveia instantaneamente o kernel de descompressão otimizado por hardware na velocidade da memória.

## 6. Query Federation Plane

O otimizador de consultas em `heraclitus-query` será estendido com uma abstração formal de Conectores Estrangeiros (*Foreign Connectors Plane*), permitindo que um único plano lógico interaja simultaneamente com múltiplos ecossistemas heterogêneos.

```
                     [ LOGICAL PLAN (Unified Query) ]
                                     │
           ┌─────────────────────────┼─────────────────────────┐
           ▼                         ▼                         ▼
[ Catalog: Postgres ]       [ Catalog: Iceberg ]      [ Catalog: Kafka ]
 (OLTP Pushdown Scan)        (Parquet File Vector)     (Streaming Window Scan)

```

### Funcionalidades do Planejador Federado

* **Predicados e Projeções Pushdown**: O otimizador analisa a subárvore da consulta federada e empurra ao máximo filtros (`WHERE`) e seleções (`SELECT`) para o motor de origem. No caso do PostgreSQL, gera-se uma query SQL nativa; no caso do S3/Iceberg, traduz-se para leituras parciais utilizando filtros de Bloom e *Zone Maps*.
* **Arrow Flight Transports**: Comunicação nativa entre nós e clusters externos utilizando fluxos de Arrow Flight, evitando completamente a serialização e desserialização de dados em trânsito.
* **Stream-to-Table Joins**: Capacidade de acoplar uma fonte de streaming de baixa latência (Kafka) a uma tabela dimensional colunar histórica (Iceberg/S3) usando janelas temporais diretamente no operador de junção física.
