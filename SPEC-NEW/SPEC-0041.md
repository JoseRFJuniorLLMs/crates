# SPEC-0040.1: HUME Core Execution Kernel (Normative Revision)

---

## 1. O Objeto Central e o Contrato de ABI: `DataChunk`

Toda a computação, manipulação e tráfego de dados nas camadas do HUME — sejam elas relacionais, vetoriais, de grafos ou de streaming — utilizam estritamente o `DataChunk` como interface binária de aplicação (ABI).

```rust
pub enum Device {
    CPU,
    GPU,
}

pub enum SelectionVector {
    Bitmap(Vec<u64>),       // Usado para alta densidade (> 25%)
    Index16(Vec<u16>),     // Usado para baixa densidade (< 25%, se fit em 64k rows)
    Index32(Vec<u32>),     // Usado para cenários de expansão ou blocos massivos
}

pub struct PhysicalRowID {
    pub segment_id: u32,
    pub page_id: u32,
    pub offset: u16,
}

pub struct DataChunk {
    pub cardinality: usize,
    pub capacity: usize,
    pub device: Device,
    pub columns: Vec<Vector>,
    pub selection: SelectionVector,
    pub validity: ValidityMask,       // Roaring Bitmap / Compact Bitmask
    pub row_ids: Vec<PhysicalRowID>,  // IDs físicos persistentes para Late Materialization
}

```

## 2. Microarquitetura Adaptativa: `SelectionVector` e `Morsel` Dinâmicos

O HUME abandona premissas estáticas de design para se ajustar ao comportamento real das consultas em tempo de execução.

### Promoção e Demora de Representação

O motor físico avalia a seletividade dos predicados após cada operador de filtragem para redefinir o layout do `SelectionVector`:

* **Alta Densidade ($\ge$ 25% de sobrevivência)**: O chunk adota a representação `SelectionVector::Bitmap`. As operações booleanas subsequentes (`AND`, `OR`, `NOT`) tornam-se operações *branchless* diretas bit a bit via instruções SIMD no processador.
* **Baixa Densidade ($<$ 25% de sobrevivência)**: O chunk converte a estrutura para `SelectionVector::Index16` ou `Index32`. Isso evita varreduras inúteis sobre milhões de bits zerados, compactando o espaço de cache L1/L2 e acelerando o operador de materialização tardia.

### Dimensionamento Dinâmico de Morsels (Adaptive Morsel Size)

O tamanho do bloco inicial de varredura não é fixo em 100k linhas. Ele inicia em **8.192 linhas** e é escalado dinamicamente para **32.768**, **65.536** ou **131.072 linhas** com base no monitoramento do `Pipeline Profiler`. O escalonador rastreia a largura em bytes das colunas ativas e a taxa de falhas de cache do processador para garantir que o `DataChunk` caiba inteiramente dentro do cache L2 ou L3 dedicado do núcleo de CPU ativo.

## 3. Filosofia de Controle de Desvios (Branching)

O HUME substitui a utopia teórica de "zero branch miss" pela diretriz pragmática de **Minimização de Desvios Imprevisíveis**.

Embora os loops computacionais internos de predicados sejam convertidos em máscaras SIMD sem desvios, o motor reconhece e isola ramificações estruturais inevitáveis (como checagem de limites de páginas, descompressão de blocos colunares, tratamento de valores nulos e loops de ordenação). O objetivo de engenharia do gerador de código JIT é garantir que desvios ocorram em padrões altamente previsíveis para o hardware.

## 4. Estrutura do Repositório de Baixo Nível: `hume-kernel`

Para garantir isolamento absoluto de hardware e desempenho de microbenchmark, os componentes primitivos são desacoplados do runtime e consolidados em um crate nativo focado em manipulação direta de memória e registradores:

```
crates/hume-kernel/
├── src/
│   ├── lib.rs
│   ├── simd/
│   │   ├── avx2.rs             # Primitivas de loop de 256-bits
│   │   ├── avx512.rs           # Primitivas de loop de 512-bits
│   │   └── neon.rs             # Primitivas ARM de 128-bits / SVE
│   ├── memory/
│   │   └── aligned_alloc.rs    # Alocador de buffers contíguos alinhados por linha de cache
│   ├── compression/
│   │   ├── bitpack.rs          # Kernels de BitPacking e FastPFor de alta velocidade
│   │   └── streamvbyte.rs      # Decodificador VarInt vetorizado
│   └── selection/
│       └── bitmap.rs           # Operações bitwise SIMD para SelectionVector::Bitmap

```

## 5. A Cadeia Alimentar dos Operadores: Prioridade Pragmática

Antes que mecanismos complexos de grafos ou buscas vetoriais sejam refinados, a fundação relacional tradicional precisa atingir o estado da arte para suportar a carga de trabalho de queries analíticas corporativas standard (TPC-H). O roadmap de engenharia do HUME é oficialmente reordenado:

```
[ Scan / Filter / Project ] ➔ [ Aggregate / Hash Join / Sort / Window ] ➔ [ Vector / Graph Primitives ]

```

A planilha e o relatório corporativo tradicional continuam sendo os predadores dominantes do ecossistema de dados. O HUME garante eficiência extrema no processamento de `GROUP BY` e `Window Functions` relacionais antes de expandir o pipeline de fusão para buscas de vizinhos mais próximos ou caminhamento de arestas.

## 6. O Protocolo de Benchmarks em Camadas

Para evitar o isolamento de métricas em testes massivos que ocultam problemas de microarquitetura, o motor será testado em três zonas distintas de estresse de hardware:

### Benchmark A: Validação de Vetorização Pura (Foco: L1/L2 Cache)

* **Volume**: 1 milhão de linhas.
* **Objetivo**: Garantir que os dados caibam inteiramente no cache local do núcleo.
* **Métricas Alvo**: Ciclos de CPU por tupla, utilização de canais SIMD e eficiência de inlining do JIT.

### Benchmark B: Validação de Banda de Memória (Foco: RAM Bandwidth)

* **Volume**: 100 milhões de linhas.
* **Objetivo**: Exceder a capacidade de cache do processador, forçando a movimentação contínua de blocos na RAM.
* **Métricas Alvo**: Taxa de transferência de bytes por segundo no barramento de memória e falhas de cache L3.

### Benchmark C: Validação de Infraestrutura de IO (Foco: Cold Storage & Prefetch)

* **Volume**: 10 bilhões de linhas.
* **Objetivo**: Forçar a leitura física de dados a partir de NVMe local ou Object Storage.
* **Métricas Alvo**: Eficiência do preditor de prefetch assíncrono via `io_uring`, velocidade de descompressão dos kernels `hume-kernel::compression` e tempo total de execução.

## 7. O Novo Roadmap Normativo de Prioridades: SPEC-0040.1

A ordem estrita de execução e congelamento de código para o desenvolvimento do ecossistema HUME passa a seguir a matriz abaixo:

1. **DataChunk ABI**: Congelamento da estrutura padrão de dados e contratos de chamadas entre operadores.
2. **Adaptive SelectionVector**: Implementação do mecanismo híbrido (Bitmap vs. Index Vector) com promoção dinâmica.
3. **Adaptive Morsel**: Framework de monitoramento de performance para reajuste automático de tamanho do bloco de linhas em tempo de execução.
4. **SIMD Kernel Layer**: Isolamento do crate `hume-kernel` e implementação dos loops primitivos por arquitetura de CPU.
5. **Late Materialization**: Operador `LateFetch` integrado com suporte à estrutura de `PhysicalRowID`.
6. **Vectorized HashJoin & Aggregation**: Desenvolvimento das primitivas de junção colunar por partições radix baseadas no tamanho de cache.
7. **Physical HUME-IR**: Construção do dialeto físico linear e passes básicos de otimização de baixo nível.
8. **Vector Operator**: Acoplamento dos kernels de busca por similaridade de embeddings (HNSW/Top-K).
9. **Graph Operator**: Acoplamento das primitivas de expansão e caminhamento estruturado de grafos.
10. **JIT Compilation Tier**: Injeção da geração de código de máquina nativo via Cranelift para eliminação de overheads residuais.

Com a aprovação desta especificação normativa, o HUME deixa de ser um ecossistema projetado sob hipóteses ideais e se torna uma máquina de processamento de dados projetada especificamente para extrair a capacidade máxima do silício real.
