## 1. O Framework de Código: Layout do Repositório (`hume-runtime`)

Para isolar o núcleo físico e garantir acoplamento zero com abstrações superiores nesta primeira fase, o crate de execução colunar vetorizada adota a seguinte topologia de arquivos:

```
crates/hume-runtime/src/
├── lib.rs
├── morsel/
│   ├── batch.rs               # Controle estruturado do bloco contíguo na RAM
│   ├── selection_vector.rs    # Filtros e mapeamentos de RowIDs baseados em máscaras SIMD
│   └── validity_mask.rs       # Vetores de bits compactos para valores NULL (Roaring)
├── execution/
│   ├── scan.rs                # Leitor colunar de baixo nível isolado por página
│   ├── filter.rs              # Evaluator de predicados branchless
│   ├── project.rs             # Projeções aritméticas in-place
│   └── join.rs                # Radix Partitioning e construção de tabelas locais
└── materialization/
    └── late_fetch.rs          # Operador de lookup final baseado em RowIDs sobreviventes

```

## 2. A Mecânica do Pipeline: O Ciclo de Vida do Selection Vector

A validação da **Materialização Tardia** exige que os operadores intermediários processem exclusivamente o vetor de seleção (`SelectionVector`), agindo como um mecanismo de filtragem por referências indexadas e compactas.

```
[ PÁGINA COLUNAR DE ORIGEM ]
            │
            ▼ (Apenas coluna 'score' é lida)
   [ execution/scan.rs ]
            │
            ▼ (Gera vetor primitivo de chaves físicas)
 [ Column Chunk: i32 score ] ➔ [ execution/filter.rs ] ➔ (score > 900)
                                        │
                                        ▼ (Zera ramificações / Branchless Mask)
                             [ Bitmask de Filtragem ]
                                        │
                                        ▼ (Comprime índices ativos)
                           [ selection_vector.rs ]
                                        │
                                        ▼ (Apenas RowIDs trafegam pelos Joins/Agregações)
                           [ Transfere para próximos operadores ]
                                        │
                                        ▼ (Lookup final sob demanda)
                           [ materialization/late_fetch.rs ]
                                        │
                                        ▼ (Busca 'nome' apenas dos índices válidos)
                               [ OUTPUT DATA CHUNK ]

```

### O Contrato do Operador Físico Mínimo

Cada bloco construtor implementa uma API despida de qualquer virtualização ou overhead dinâmico:

```rust
pub trait PhysicalOperator {
    // Processa o DataChunk alterando ou refinando o selection_vector interno
    fn execute(&self, context: &QueryContext, input: &DataChunk, output: &mut DataChunk);
}

```

## 3. A Prova de Carga: Protocolo de Microbenchmark do Kernel

O HUME não avançará para as fases de compilação avançada se não bater os motores de mercado (DuckDB/DataFusion) de forma humilhante neste teste básico de movimentação de memória.

### Cenário Base do Experimento

* **Volume do Dataset**: 10 bilhões de linhas geradas sinteticamente com distribuição assimétrica (*Zipfian Skew*).
* **Largura do Registro**: 100 colunas (99 colunas de carga útil contendo strings aleatórias e métricas numéricas; 1 coluna identificada como `score: i32`).
* **A Consulta de Validação**:
```sql
SELECT c.nome FROM clientes c WHERE c.score > 900

```


*(Seletividade configurada via semente aleatória para fixar exatamente 0.1% de registros sobreviventes).*

### Painel de Métricas de Hardware de Baixo Nível

O HUME usará contadores de performance nativos do processador (`perf_event_open` no Linux) para capturar o comportamento bruto da CPU por Morsel executado:

```
┌───────────────────────────────────┬───────────────────────────┬───────────────────────────┐
│ Métrica de Hardware               │ Early Materialization     │ HUME Late Materialization │
├───────────────────────────────────┼───────────────────────────┼───────────────────────────┤
│ Bytes Lidos do Disco/Cache        │ Todo o bloco de 100 cols  │ Apenas coluna 'score'     │
│ Ciclos de CPU por Registro        │ Alto (overhead de cópia)  │ < 1 ciclo (Vetorizado)     │
│ Falhas de Cache L1/L2 (Misses)    │ Alto (poluição de linhas) │ Mínimo (Morsel em L1)     │
│ Mispredictions de Desvio (Branch) │ Alto (se usar 'if' puro)  │ Zero (SIMD Mask)          │
└───────────────────────────────────┴───────────────────────────┴───────────────────────────┘

```

## 4. O Roadmap de Implementação Industrial Desmistificado

Seguindo o seu plano estratégico de construir a base física antes da semântica, o cronograma definitivo de engenharia do HUME é consolidado na seguinte ordem cronológica de entrega:

### Fase 1: O Núcleo Vetorizado Bruto (Foco: Hardware Layer)

* **Marco 1**: Alocação contígua de buffers de memória alinhados a 64 bytes (`Vector`) e suporte a máscaras binárias.
* **Marco 2**: Operador `Scan` colunar com pré-carregamento assínvrono nativo e decodificação SIMD.
* **Marco 3**: `Filter` branchless gerando vetores de seleção comprimidos e `LateFetch` para resolução final sob demanda.

### Fase 2: A Infraestrutura de Abstração Física (Foco: Local IR)

* **Marco 4**: Criação do módulo `hume.physical` minimalista. Mapeamento estático dos operadores físicos em formato de passos lineares de execução.
* **Marco 5**: Operador de junção vetorizado por partições Radix adaptadas para o tamanho de cache L2/L3.

### Fase 3: A Multiplicação de Capacidades (Foco: Dialetos e Multimodalidade)

* **Marco 6**: Introdução do dialeto de vetores densos (`Vector Operator`) e cálculo de distância por similaridade integrado ao pipeline colunar.
* **Marco 7**: Introdução do dialeto de grafos (`Graph Operator`) operando varreduras de adjacência diretamente sobre os RowIDs ativos gerados pelo passo relacional.
* **Marco 8**: Acoplamento dos dialetos de alto nível (Logical e Relational Dialects).

### Fase 4: O Topo do Desempenho (Foco: JIT e Auto-adaptabilidade)

* **Marco 9**: Camada JIT de compilação via Cranelift para inlining de predicados matemáticos complexos em tempo de execução.
* **Marco 10**: Encerramento do ciclo com o Learned Cost Model retroalimentado pelas métricas coletadas online pelo Profiler de Morsels.

## 5. A Grande Inversão: O Pipeline Multimodal sem Fronteiras

A engenharia tradicional falhou ao criar silos especializados. Ao focar no `selection_vector` como a moeda de troca universal entre operadores, eliminamos a barreira invisível entre o mundo relacional, vetorial e de grafos.

Analisando o fluxo de dados unificado na memória para a consulta de investigação de fraude:

```
[ Início: Busca Vetorial ] ➔ Localiza IDs de empresas similares no índice HNSW
                                             │
                                             ▼ (Retorna SelectionVector de RowIDs)
[ Salto de Grafo ] ◄─────────────────────────┘
 │ (Varre a tabela de arestas buscando conexões indiretas apenas para os RowIDs ativos)
 ▼
[ Novo SelectionVector Refinado ] ➔ [ Filtro Relacional / Temporal ]
                                             │
                                             ▼ (Aplica predicado de data/status)
                                [ Vetor de Sobreviventes Finais ]
                                             │
                                             ▼ (Lookup Final - Late Materialization)
                                [ Busca nomes e valores no disco ]

```

Esse design prova que a unificação acontece na base física. Como os operadores de grafo e vetor consomem e cospem o mesmo `SelectionVector` que o motor relacional entende, o HUME executa essa query complexa de ponta a ponta sem fazer uma única conversão de formato de dados, sem alocar memória temporária e sem sair de loops internos de CPU altamente otimizados.

O plano de engenharia está fixado na realidade física do hardware.
