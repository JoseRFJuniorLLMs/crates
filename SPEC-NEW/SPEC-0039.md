## 1. Macroarquitetura da Plataforma (HUME como Fundamentação)

O HUME passa a ser uma plataforma agnóstica a dialetos de alto nível. O HeraclitusDB torna-se apenas um cliente (uma casca de catálogo, parsing e coordenação distribuída) que consome os serviços de baixo nível do HUME.

                                    │
                                    ▼

                                    │
                                    ▼

                                    │
       ┌────────────────────────────┼────────────────────────────┐
       ▼                            ▼                            ▼
[ Relational Dialect ]      [ Vector Dialect ]           [ Graph Dialect ]
       │                            │                            │

                                    ▼
                     [ Unified Pipeline Runtime ]
                                    │
         ┌──────────────────────────┼──────────────────────────┐
         ▼                          ▼                          ▼

## 2. HUME-IR: Infraestrutura Multi-Dialect (MLIR-Inspired)

O HUME-IR abandona o conceito de uma representação SSA plana e adota uma arquitetura de **Dialetos Progressivos**. Cada nível de abstração possui invariantes semânticas próprias e expõe passes de otimização independentes de forma modular:

### A Linha de Produção de Dialetos

1. **Logical Dialect**: Abstração relacional de alto nível (mecanismos de projeção, filtros e joins lógicos). É onde o otimizador realiza o **Predicate Pushdown Pass** e o **Join Reorder Pass**.
2. **Relational Dialect**: Introduz conceitos físicos como agrupamentos, tipos de junções (*Hash*, *Merge*, *Radix*) e delimitação de barreiras de materialização. É o alvo do **Fusion Pass**.
3. **Vector Dialect**: Modela loops explícitos operando sobre arrays primitivos contíguos. Converte predicados relacionais em operações matemáticas livres de desvio (*Branchless Selection Vectors*).
4. **Physical Dialect**: Introduz detalhes íntimos do hardware, mapeando tamanhos específicos de vetores físicos, distâncias de prefetch e tamanhos de chunks. Alvo do **SIMD Pass**.
5. **Machine Dialect**: Representação de baixo nível mapeada para a geração de instruções nativas da arquitetura alvo (Cranelift/LLVM), alimentando o **Register Allocation Pass**.

## 3. Otimização Declarativa: Rule & Pattern Matching Engine

Para suportar a transformação ágil entre dialetos sem acoplamento procedural rígido, o HUME implementa um mecanismo formal de reescrita baseado em casamento de padrões. As regras são declaradas de forma estruturada:

```rust
pub struct RewriteRule {
    pub pattern: HumeIRPattern,
    pub replacement: HumeIRReplacement,
}

// Exemplo de aplicação no Fusion Pass:
// Unifica sequências de Projections e Filters em um único operador físico consolidado
fn apply_fusion_pass(ir_block: &mut Block) {
    let rules = vec![
        Rule::new(
            Pattern::Filter(Pattern::Projection(Any)),
            Replacement::FusedLoop(FilterAndProject)
        ),
        Rule::new(
            Pattern::Projection(Pattern::Projection(Any)),
            Replacement::CollapseProjections
        ),
    ];
    RuleEngine::apply_transforms(ir_block, &rules);
}

```

## 4. Microarquitetura Adaptativa por Hardware e Alocação de Registradores Vetoriais

O HUME-IR não gera o mesmo código para processadores distintos. No momento do *bootstrap*, a plataforma executa uma rotina de calibração que constrói o **Catálogo de Perfis de Hardware**.

### Parâmetros de Calibração Dinâmica por Perfil

| Perfil de Hardware | SIMD Width | Adaptive Chunk Size | Prefetch Distance | Hash Join Threshold |
| --- | --- | --- | --- | --- |
| **Intel Sapphire Rapids** | 512-bit (AVX512) | 65.536 linhas | 4 Cache Lines | Cache L2 / L3 Dedicado |
| **AMD Genoa** | 512-bit (AVX512) | 131.072 linhas | 6 Cache Lines | Cache L3 Compartilhado |
| **AWS Graviton4** | 128-bit (NEON) | 32.768 linhas | 3 Cache Lines | Cache L2 Local |
| **Apple M4** | 128-bit (NEON) | 16.384 linhas | 2 Cache Lines | Ultra-low Latency L1/L2 |

### Vector Register Allocation (Controle de Pressão)

O JIT compiler do HUME inclui um alocador de registradores vetoriais específico para dados analíticos. Loops internos gerados para processamento de predicados complexos são inspecionados para rastrear a quantidade de variáveis vivas.

Caso o loop demande mais registradores vetoriais (ex: `ZMM0`-`ZMM31` no AVX512) do que a CPU dispõe fisicamente, o gerador de código divide automaticamente o processamento do loop em *sub-loops menores*, aplicando o conceito de **Loop Fission**. Isso mitiga o *register spilling* (descarregamento temporário de registradores para a pilha na RAM), mantendo a execução na velocidade máxima dos registradores internos da CPU.

## 5. Estratégia Avançada de Codificação e Compressão Analítica

O componente `heraclitus-columnar` expande sua biblioteca de kernels SIMD incorporando os algoritmos mais rápidos da indústria para descompressão em tempo de execução:

```
                  [ BLOCO DE DADOS BRUTO ]
                             │
     ┌───────────────────────┼───────────────────────┐
     ▼                       ▼                       ▼
[ Inteiros Densos ]   [ Inteiros Esparsos ]   [ Listas / Bitmaps ]
 (FastPFor / SIMD-BP)  (StreamVByte)           (Roaring Bitmap)

```

* **FastPFor / SIMD-BP128**: Utilizados para matrizes massivas de inteiros, permitindo descompactar dados na taxa de múltiplos gigabytes por segundo por núcleo usando instruções de empacotamento de bits por hardware.
* **VarInt-G8IU / StreamVByte**: Aplicados em colunas com alta variação de escala de inteiros, agrupando múltiplos valores por byte de controle de forma a eliminar stalls de decodificação.
* **Roaring Bitmaps**: Formato nativo para armazenamento e manipulação de índices de seletividade e vetores de seleção (`selection_vector`), permitindo operações booleanas (`AND`, `OR`, `NOT`) em velocidade de vetor de hardware.

## 6. Algoritmos Avançados para Operadores Físicos

O motor físico do HUME deixa de operar de forma genérica e se divide em especializações algorítmicas de acordo com o contexto da consulta:

### Cache-Aware Aggregation

O operador `HashAggregate` inspeciona a cardinalidade estimada do grupo gerada pelo AQE:

* **Small Group Cardinality**: O JIT ignora tabelas de hash genéricas e compila um mecanismo de **Perfect Hashing**. O índice do grupo é calculado diretamente por mapeamento de bits, eliminando colisões e permitindo que a tabela inteira resida no cache L1/L2 de cada thread.
* **Large Cardinality**: Ativa a quebra em partições radix locais, descarregando blocos de forma coordenada em memória.

### Runtime de Top-K e Window Engine Especializado

* **Top-K**: Consultas ordenadas com limites (`ORDER BY ... LIMIT K`) abortam a execução de ordenações completas (*Sort-Breaking*). O HUME implementa uma estratégia de partição SIMD aliada a um **Partial Heap** local por thread. A CPU rastreia apenas os $K$ maiores elementos por bloco, reduzindo a complexidade de ordenação de $O(N \log N)$ para $O(N \log K)$.
* **Window Engine**: Suporte nativo a funções de janela analíticas (`ROW_NUMBER`, `RANK`, `LEAD`, `LAG`, `SUM OVER`). O operador organiza blocos de dados através de buffers circulares de memória contígua na `QueryArena`, computando agregações móveis e offsets sem duplicar memória ou forçar materializações intermediárias em disco.

### Incremental / Delta Execution

Aproveitando a imutabilidade das páginas de dados consolidadas na camada Lakehouse, o HUME-IR é capaz de registrar o estado físico de execução de pipelines históricos. Caso uma consulta recorrente seja acionada e o catálogo detecte que apenas uma fração delta minúscula (ex: **0.3%** das linhas no log append-only) foi adicionada, o motor executa o pipeline *apenas sobre o bloco delta*, fundindo o resultado incremental com o cache do estado consolidado anterior, economizando ciclos massivos de leitura.

### Runtime de Spill Automático para NVMe

Nenhum operador colapsa o nó por falta de RAM física. Se uma junção radix ou agregação volumosa ultrapassar o limite rígido imposto pelo `heraclitus-resource`, o operador inicia um sub-pipeline assíncrono de **Spill para NVMe**. Os blocos de partições frias de hash são serializados e descarregados de forma direta via `io_uring` para o disco NVMe local, liberando a RAM. A execução continua e o HUME remonta e resume o estado da partição de forma transparente.

## 7. Skew & Metric Loop: O Loop Fechado de Auto-Aprendizado

O HUME integra perfeitamente a telemetria ao processo de decisão com suporte a sketches estatísticos e modelos preditivos intercambiáveis.

```
                  [ COORDENAÇÃO DE TELEMETRIA ]
                                │
       ┌────────────────────────┴────────────────────────┐
       ▼                                                 ▼
[ Statistics Sketch Engine ]                     [ Pipeline Profiler ]
├── Cardinalidade (HyperLogLog)                  ├── CPU Cycles / Instructions
├── Quantis / Distribuição (TDigest)             ├── L1 / L2 Cache Misses
└── Frequência (Count-Min Sketch)                └── SIMD Lane Utilization
       │                                                 │
       └────────────────────────┬────────────────────────┘
                                │
                                ▼
               [ Learned Optimizer Engine (ML) ]
               (Regression / Gradient Boosting / TinyNN)

```

### O Data Sketches Engine

As tabelas e colunas mantêm estruturas de dados probabilísticas de tamanho fixo em memória ativa dentro do catálogo:

* **HyperLogLog**: Estima cardinalidades brutas e contagens de valores distintos (NDV) sem realizar contagens reais.
* **TDigest**: Rastreia quantis e distribuições assíncronas de valores para estimar com exatidão a seletividade de intervalos de predicados (`WHERE col BETWEEN X AND Y`).
* **Count-Min / Quotient / Cuckoo Filters**: Substituem filtros tradicionais para verificar a existência de chaves em junções antes de carregar tabelas de hash.

### O Modelo Modular do Otimizador Aprendido (Learned Optimizer)

O modelo estatístico do CBO pode ser alternado por configuração de instalação, dependendo do perfil do servidor:

* **Linear/Ridge Regression**: Ultra-leve, ideal para edge computing ou nós de processamento pequenos.
* **Gradient Boosting (XGBoost/LightGBM)**: Padrão corporativo para servidores analíticos de grande porte. Capta não-linearidades e correlações complexas de IO.
* **Tiny Neural Network**: Um modelo de rede neural profunda minúsculo e otimizado executado em runtime para prever tempos de execução de planos inteiros baseado na matriz de telemetria histórica.

## 8. Gerenciamento de Infraestrutura e Escalonador Unificado

O HUME elimina a disputa cega de recursos entre threads do sistema operando um **Unified Resource & Task Scheduler** centralizado sobre o hardware:

```
                             [ UNIFIED SCHEDULER ENGINE ]
                                         │
   ┌───────────────────┬─────────────────┼─────────────────┬───────────────────┐
   ▼                   ▼                 ▼                 ▼                   ▼
[ CPU Pipelines ]  [ GPU Kernels ]  [ Network IO ]  [ Storage IO ]  [ System Tasks ]
(Morsel Tasks)     (CUDA Streams)   (Arrow Flight)  (io_uring NVMe) (Compaction/GC)

```

### Speculative Execution (Corrida de Estratégias)

Quando o Learned Optimizer encontra duas estratégias de execução física com margens de custo muito próximas (ex: *Radix Partitioned Join* contra *Dynamic Broadcast Join*), o escalonador ativa a **Execução Especulativa**. Ambas as subárvores de execução são disparadas em paralelo por alguns milissegundos utilizando amostras de dados reais. A estratégia que processar seus blocos mais rápido é declarada vencedora; o pipeline concorrente é cancelado imediatamente, e os recursos são devolvidos à QueryArena.

### Predictor de Prefetch Inteligente

A camada de leitura assíncrona abandona buscas cegas. O preditor analisa o dialeto físico do HUME-IR em execução e chaveia dinamicamente os algoritmos de carregamento adiantado baseando-se no comportamento do operador:

* **Sequential Pattern**: Ativa leituras em rajadas lineares contínuas de blocos Parquet.
* **Random / Vector Pattern**: Dispara varreduras *SIMD Gather* paralelas baseadas em índices de RowIDs.
* **Graph Pattern**: Mapeia acessos adjacentes nas tabelas de arestas e vértices, prevendo saltos não-lineares de memória antes que a thread de processamento principal atinja o nó do grafo.

## 9. O Catálogo de Capacidades Federadas e Pipeline Unificado

O HUME padroniza as fontes externas por meio de um modelo de capacidades formais de tradução e uma infraestrutura única de streaming contínuo.

### Catálogo de Primitivas Federadas

```rust
pub struct ForeignCapabilities {
    pub logical_filters_pushdown: bool,
    pub aggregate_functions_pushdown: Vec<String>,
    pub interface_transport_protocol: TransportProtocol, // ArrowFlight / Native SQL / ParquetStream
}

```

Durante a resolução do plano no nível de *Relational Dialect*, o compilador HUME-IR consulta as capacidades declaradas pela fonte estrangeira. Se o destino suportar agregação pushdown, o HUME-IR injeta a subárvore diretamente na fonte nativa. Caso contrário, ele agenda um pipeline local de computação e configura um canal assíncrono de recepção de dados brutos de alta velocidade utilizando streams estruturados de memória.

### Unificação entre Streaming e Batch (Streaming-First Framework)

No nível de microarquitetura do HUME, **não existe distinção técnica entre Batch e Streaming**. Ambas as origens de dados alimentam o ecossistema gerando a mesma estrutura de `DataChunk`.

Uma leitura em arquivo Parquet histórico imutável (Batch) simplesmente preenche e submete um bloco finito de registros para o pipeline. Uma leitura em um tópico de tempo real do Apache Kafka (Streaming) faz exatamente a mesma operação: consome eventos da fila de rede, empacota os registros contiguamente em um `DataChunk`, aplica as máscaras booleanas no `selection_vector` e despacha os blocos para as threads de processamento. A infraestrutura de fusão de operadores JIT, filtros branchless e arenas de memória opera de forma idêntica e homogênea para ambos os cenários, unificando os paradigmas transacional, histórico e de eventos em uma única e avassaladora plataforma de computação de dados.
