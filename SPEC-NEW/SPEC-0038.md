# SPEC-0038: HUME (Heraclitus Unified Micro-Execution Engine)

---

## 1. O Elo Perdido: Unified Execution IR (SSA-Form)

Para desacoplar completamente a inteligência do planejador físico da geração de código e permitir otimizações cross-paradigma (relacional, grafos e vetores), o Heraclitus introduz o **HUME-IR**, uma representação intermediária em formato *Static Single Assignment* (SSA) linear e fortemente tipada.

```
       [ Physical Plan (Relational / Graph / Vector / Text) ]
                                 │
                                 ▼
                     [ HUME-IR (SSA Builder) ]
                                 │
     ┌───────────────────────────┴───────────────────────────┐
     ▼ (Otimizações de Baixo Nível)                          ▼ (Pipeline Fusion)
[ Dead Code Elimination ]                               [ Loop Inlining & Masking ]
     │                                                       │
     └───────────────────────────┬───────────────────────────┘
                                 │
                                 ▼
                 [ HUME-IR Adaptive Code Gen Plane ]
                                 │
   ┌──────────────────────┬──────┴───────────────┬──────────────────────┐
   ▼                      ▼                      ▼                      ▼
[ Vector Interpreter ] [ Cranelift JIT ]      [ LLVM Compiler ]     [ GPU Cuda/HIP ]
(Cold Tiers)           (Hot Tiers)            (Static Compiles)     (Massive Blocks)

```

O HUME-IR elimina o acoplamento com backends específicos. Se o plano físico precisa executar uma filtragem multimodal, ele gera blocos de instruções HUME-IR. Essa IR pode ser enviada para o interpretador de vetores, compilada instantaneamente com Cranelift ou traduzida para kernels de GPU, garantindo portabilidade absoluta de performance.

## 2. A Estrutura Física de Fluxo: Vectorized Data Chunks

O motor abandona qualquer granularidade baseada em tuplas ou arrays puros dentro do pipeline de execução. A unidade fundamental de dados que trafega entre os operadores físicos é o `DataChunk`.

```rust
pub struct Vector {
    data_ptr: *mut u8,             // Ponteiro alinhado para memória contígua (SIMD-ready)
    validity_mask: *mut u64,       // Bitmask colunar para controle de valores NULL
    data_type: DataType,           // Tipo nativo do vetor (i32, u64, StringRef, etc.)
}

pub struct DataChunk {
    columns: Vec<Vector>,          // Vetores contíguos alinhados por tipo
    selection_vector: *const u16,  // Array de IDs ativos (Late Materialization / Filter Masks)
    cardinality: usize,            // Número de registros ativos no chunk (padrão: 102.400)
    capacity: usize,               // Capacidade física alocada para o bloco
}

```

Cada operador físico implementa a assinatura estrita: `fn execute(input: &DataChunk, output: &mut DataChunk)`. Não há alocação de memória, *boxing* de tipos ou cópia de memória (`memcpy`) entre operadores adjacentes de um mesmo pipeline.

## 3. Pipeline Fusion & Adaptive Multi-Tiered JIT

O otimizador de IR analisa cadeias de operadores unificando-os em blocos de execução contínuos. Em vez de passar o `DataChunk` do Scan para o Filter, e deste para a Projection, o HUME-IR realiza o **Operator Fusion** gerando um único loop nativo em memória.

### Modelo de Compilação Adaptativa (JVM-style para Dados)

O ciclo de vida de execução de uma instrução HUME-IR é dinâmico e monitorado por amostragem de frequência de execução (*hotness profiling*):

```
[ HUME-IR Block ] ➔ (Execução Inicial) ➔ [ 1. Vector Interpreter ] (Zero compile overhead)
                         │
                         ▼ (Morsels Processados > 10)
                 [ 2. Hot Tier: Cranelift Baseline JIT ] (Loop inlining básico)
                         │
                         ▼ (Morsels Processados > 100)
                 [ 3. Super Hot Tier: Vectorized JIT (Cranelift/LLVM) ] (SIMD Auto-vectorization)

```

Filtros analíticos pesados e projeções matemáticas complexas atingem o nível *Super Hot* em poucos milissegundos, gerando instruções de máquina nativas especializadas para o layout exato das páginas consultadas.

## 4. Late Materialization & Branchless Runtime SIMD

O Heraclitus adota a **Materialização Tardia**. O operador de varredura inicial (`Scan`) lê exclusivamente as colunas presentes nos predicados de filtragem mais seletivos e a coluna de chaves primárias virtuais (`RowIDs`).

```
[ Scan Coluna A ] ➔ [ Predicate Filter ] ➔ [ Generates SelectionVector (RowIDs) ]
                                                       │
                                                       ▼
[ Late Materialization Operator ] ◄────────────────────┘
 │ (Lê apenas as linhas ativas das colunas B e C via SIMD Gather)
 ▼
[ Final Projection / Output Chunk ]

```

As operações de filtragem e computação dentro do loop JIT utilizam uma estratégia **Branchless** (livre de desvios condicionais) para evitar o colapso da tabela de predição de saltos da CPU (*Branch Prediction Explosion*).

```
// Abordagem Tradicional com Branch (Ineficiente)
if data[i] > 100 { result[write_idx] = data[i]; write_idx += 1; }

// Abordagem HUME SIMD Branchless (State-of-the-Art)
let mask = simd_gt_i32(data_vector, 100);
let compressed_vector = simd_compress(data_vector, mask);
simd_store(result_ptr, compressed_vector);

```

### Kernels SIMD Especializados por Perfil de Hardware

O módulo `heraclitus-columnar` expõe assinaturas especializadas por arquitetura detectada no *boot* do sistema através do **Execution Profile**:

| Kernel Especializado | Target x86_64 (Moderno) | Target x86_64 (Legado) | Target ARM64 (Sovereign Cloud) |
| --- | --- | --- | --- |
| `simd_eq_i32` | `_mm512_cmpeq_epi32_mask` | `_mm256_cmpeq_epi32` | `vceqq_s32` (NEON) / SVE Mask |
| `simd_contains_str` | AVX512 Vectorized String | AVX2 SWAR Search | NEON Vectorized Mask |

## 5. Ciclo de Vida da Memória: Query Arenas & Alocação NUMA-Aware

O Heraclitus elimina o uso de alocadores globais (`malloc`, `jemalloc` ou `Vec::new()`) durante a execução de consultas analíticas. Cada consulta cria uma estrutura hierárquica isolada de **Memory Arenas** vinculadas à topologia NUMA do servidor.

```
       [ NUMA Node 0 (Sockets / L3 Cache Local) ]            [ NUMA Node 1 (Sockets / L3 Cache Local) ]
                           │                                                     │
        ┌──────────────────┴──────────────────┐               ┌──────────────────┴──────────────────┐
        ▼                                     ▼               ▼                                     ▼
[ Work-Stealing Thread 0 ]         [ Work-Stealing Thread 1 ] [ Work-Stealing Thread 2 ]         [ Work-Stealing Thread 3 ]
        │                                     │               │                                     │
        ▼                                     ▼               ▼                                     ▼
[ QueryArena Local Node 0 ]         [ QueryArena Local Node 0 ] [ QueryArena Local Node 1 ]         [ QueryArena Local Node 1 ]
        │                                     │               │                                     │
        ▼                                     ▼               ▼                                     ▼
[ Radix Partition Table 0 ]         [ Radix Partition Table 1 ] [ Radix Partition Table 2 ]         [ Radix Partition Table 3 ]
        │                                     │               │                                     │
        └──────────────────┬──────────────────┘               └──────────────────┬──────────────────┘
                           ▼                                                     ▼
              [ Local Merge Join (Node 0) ]                         [ Local Merge Join (Node 1) ]

```

* **Estratégia QueryArena**: Toda memória intermediária para agregação, buffers de ordenação e tabelas de hash locais é alocada em blocos contíguos de 4MB diretamente na `QueryArena` ligada ao nó NUMA onde a thread está rodando. Ao término da query, executa-se um comando global `arena.reset()`, reduzindo o custo de desalocação a zero ($O(1)$).
* **Work-Stealing Morsel Scheduler**: Se a thread 3 esvaziar sua fila local de blocos (*Morsels*), ela rouba um morsel de dados pendente de outra thread. No entanto, o escalonador prioriza o roubo de threads contidas no *mesmo nó NUMA* para mitigar o tráfego de interconexão de barramento de memória externa (UPI/XGMI).

## 6. Radix Hash Join Engine

Junções tradicionais criam uma única tabela de hash gigante em memória, gerando acessos aleatórios severos que quebram o cache L3 da CPU ao processar o lado de busca (*probe side*). O Heraclitus implementa o **Radix Hash Join**.

```
[ Input Chunk (Build Side) ] ➔ [ Radix Partitioning Step ] ➔ Quebra dados em subpartições
                                                                     │
                                                                     ▼
[ Probe Side Search ] ◄─────── [ Cache-Sized Hash Tables ] ◄─────────┘
 (Acessos ordenados locais)     (Garante encaixe total no cache L2/L3 por thread)

```

1. **Fase de Particionamento Radix**: As chaves de junção são inspecionadas e clusterizadas em subpartições baseadas nos bits mais significativos do hash da chave.
2. **Isolamento em Cache**: O tamanho de cada partição radix resultante é calibrado para se ajustar inteiramente ao cache L2 ou L3 do núcleo de processamento.
3. **Build & Probe Locais**: O operador constrói mini-tabelas de hash locais por partição e executa a varredura sem gerar nenhuma falha de cache de memória principal.

## 7. Auto-Tuning Closed Loop: Estatísticas Online e Learned Cost Model

O subsistema estatístico e o Otimizador Baseado em Custos (CBO) passam a operar em um loop de retroalimentação contínuo. O Heraclitus deixa de confiar puramente em histogramas estáticos históricos.

```
[ CBO Model Estimate ] ➔ [ Runtime Execution Engine ] ➔ [ Telemetry Tracker ] (Coleta dados reais)
         ▲                                                               │
         │                                                               ▼
 [ Adaptive Updates ] ◄────── [ Learned Cost Model (Regression) ] ◄──────┘

```

### Estatísticas Online (`Statistics Engine`)

Durante a passagem dos `DataChunks` pelos operadores físicos, o framework de observabilidade intercepta a cardinalidade real e a taxa de seletividade dos predicados (`cardinality` real versus estimada). Essas métricas atualizam imediatamente as estatísticas em memória viva no catálogo.

### Modelo de Custo Aprendido (Learned Cost Model)

A estimativa de custo de um plano passa a utilizar um modelo de regressão linear dinâmica cujos coeficientes são otimizados pelo histórico de telemetria da máquina:

$$\mathcal{C}_{\text{estimated}} = \beta_0 + \beta_1 \cdot X_{\text{rows}} + \beta_2 \cdot X_{\text{memory\_footprint}} + \beta_3 \cdot X_{\text{io\_ops}}$$

Se o sistema operacional reporta sistematicamente que leituras em nuvem (S3/Iceberg) possuem latências maiores do que as estimadas originalmente, o modelo ajusta o peso incremental do coeficiente $\beta_3$, forçando o CBO a autopunir planos com alto tráfego de rede em consultas futuras, atingindo o comportamento de *Self-Tuning*.

## 8. Abstração Total de IO Assíncrono

O motor de pipelines elimina chamadas blocantes de leitura. A ingestão e leitura do sistema colunar operam por meio de uma fila unificada de submissão orientada a eventos assíncronos (apoiada em interfaces de alto desempenho como `io_uring` no Linux).

```
[ Morsel Execution Execution ] ➔ Dispara Solicitação ➔ [ Prefetch Queue Manager ]
                                                               │
                                                               ▼
[ Physical Pipeline ] ◄─── [ Returns Futures ] ◄─── [ io_uring / NVMe Direct IO ]

```

As threads de computação do pipeline analítico nunca entram em estado de espera (*sleep*) aguardando dados de disco ou rede. Elas processam os chunks ativos na RAM enquanto a infraestrutura assíncrona garante o fluxo contínuo de dados das próximas páginas.

## 9. O Mecanismo de Execução Multimodal Unificado

A unificação dos quatro paradigmas suportados pelo HeraclitusDB (relacional, grafos, texto completo e vetores densos) é alcançada na base do ecossistema de vetores. Cada modelo de dados expõe suas estruturas de forma que possam ser consumidas diretamente por operadores homogêneos que aceitam `DataChunks`.

```
                        [ UNIFIED VECTOR PIPELINE ]
                                     │
      ┌──────────────────────┬───────┴──────────────┬──────────────────────┐
      ▼                      ▼                      ▼                      ▼
[ Relational Operators ] [ Graph Operators ]   [ Vector Operators ]   [ Text Operators ]
 - HashJoin               - Breadth-First Scan  - HNSW Top-K           - BM25 Token Search
 - Project                - Pattern Match       - Cosine Distance      - Inverted Index Match
      │                      │                      │                      │
      └──────────────────────┼──────────────────────┴──────────────────────┘
                             ▼
               All implement: `Operator::execute(chunk)`
                             │
                             ▼
             [ Cohesive Pipeline Fusion Loop ]

```

### O Catálogo de Capacidades Federadas (*Capabilities Catalog*)

Para realizar pushdowns inteligentes e otimizados em fontes remotas na federação de consultas, o catálogo mapeia explicitamente as matrizes primitivas de capacidade computacional de cada conector:

```rust
pub struct StorageCapabilities {
    pub supports_filter_pushdown: bool,
    pub supports_aggregation_pushdown: bool,
    pub supported_simd_encodings: Vec<CompressionType>,
    pub native_projection_limit: usize,
}

```

O planejador decompõe a árvore lógica avaliando este catálogo de capacidades. Se o destino for o Apache Iceberg, ele extrai dados de metadados nativos (*Zone Maps* e filtros de Bloom). Se o destino for o PostgreSQL remoto, ele injeta expressões complexas de agregação diretamente via dialeto SQL nativo da fonte, poupando a banda de rede do cluster principal.

### Unificação entre Streaming e Batch

Os conectores de streaming de dados contínuos (como Apache Kafka) deixam de ser tratados como integradores externos isolados de ETL. O fluxo do Kafka alimenta o mecanismo de execução injetando registros diretamente formatados como `DataChunks` contínuos de memória.

O pipeline de execução física analítica consome esses blocos sem discernir se a origem é um arquivo Parquet histórico imutável (Batch) ou um fluxo de eventos em tempo real (Streaming), unificando em uma única arquitetura de execução as vantagens de motores *streaming-first* com a performance massiva de bancos colunares de última geração.
